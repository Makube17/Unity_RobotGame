# Change Log

## 2026-06-13

- Disabled the legacy crane-style hand motion by default via `HandMoveController.enableLegacyCraneMotion = false`.
- When legacy crane motion is disabled, `HandMoveController.MoveAndGrasp` now passes only the grasp yaw to `RobotHandController` and skips root movement, gripper open/close, and pseudo object attachment.
- `ScanManager.ExecuteGrasp` now sends the grasp yaw directly to `RobotHandController` unless legacy crane motion is explicitly enabled.
- Added Inspector-visible grasp yaw receive diagnostics to `RobotHandController`: received flag, normalized yaw, receive count, source, and time.
- Added Console logs confirming when the yaw is passed and received.
- Replaced the default grasp success check with a finger-contact based judge plus lift validation: both finger probes can be required to touch the same object, the target is lifted only after contact is accepted, and the lift fails if it hits another object or rotates too much.
- Added `HandMoveController.enableSimpleSimulatedMotion`, enabled by default, so the hand moves to the grasp candidate and closes without using the legacy pseudo-attach crane behavior.
- Added `HandMoveController.motionRoot` so simple simulated motion can move only the hand/wrist target instead of the robot base, and added `ScanManager.simpleHandYawCorrectionDeg = -90` degrees for the simple simulated grasp direction.

## 2026-06-20

- Localized player-facing UI text in `ScanManager` from mixed English/Japanese to Japanese while leaving debug log messages unchanged.
- Made the grasp judge more game-friendly for submission testing: widened the finger contact radius, added a default-on contact-box fallback, enlarged the contact box, and made lift blocker failure optional.
- Simplified the catch result display to show earned points and total score instead of target name, lift height, and rotation amount.
- Removed the attempt count line from the catch result panel.
- Changed scoring from rank-based fixed points to Q-based risk/reward points: higher success rates give fewer points, lower success rates give more points.
- Added `GraspGeometryReporter` as a separate measurement/debug script for hand transforms, finger/contact distances, collider shapes, object prefab bounds, and spawned object bounds.
- Extended `GraspGeometryReporter` with M9-origin experiment-coordinate reporting and collider-bounds based finger body center reporting for Blender-origin-offset hand parts.
- Added a fixed-base grasp motion to `RobotHandController` and made `ScanManager` prefer it over whole-hand transform motion when enabled.
- Adjusted the fixed-base grasp motion order so the hand finishes aiming, waits briefly, then closes; the post-close lift motion is now optional and disabled by default.
- Added a finger-tip point based grasp judge to `ScanManager`: success now checks the midpoint between the left/right finger tip transforms after motion, compares it with the target object's bounds center, and records Inspector-visible distance diagnostics. Legacy candidate-center fallback is disabled by default for this judge.
- Fixed a mismatch where the gripper could close and still run success judgement even when the fixed-base elbow joints were not assigned. Fixed-base motion now aborts with a failure reason, and `ScanManager` rejects the grasp before contact judgement when fixed-base motion did not complete.
- Tightened the finger-tip judge by requiring the finger-tip midpoint to stay near the selected candidate position, preventing a stationary hand from succeeding against an unrelated nearby object.
- Added `RobotHandController.fixedBaseExperimentXOffsetMm` so fixed-base M9 pitch can be adjusted in experiment-coordinate X before angle calculation. The fixed-base debug log now reports raw/adjusted target coordinates and raw/clamped pitch X values for height tuning.
- Changed AI candidate world-position handling in `ScanManager`: depth-based camera projection is now the default source for grasp positions, while Raycast is only a fallback. Grasp-position offsets and marker-visual offsets are now separated so display markers can be lifted without changing the actual robot target.
- Added `CandidateDepthValueMode` to `ScanManager` so AI candidate depth can be interpreted as meters, normalized 0-1 depth, or inverted normalized 0-1 depth. The default matches the current `DepthCapture.shader`, which outputs near objects as larger values.
- Added `ScanManager.candidateMarkerLiftY` as a simple world-Y-only display lift for AI candidate markers, reducing confusion with depth offsets that move points along the camera ray.
- Reworked candidate height handling around a surface/grasp/marker split: depth/Raycast now produces a surface point, `candidateGraspSurfaceOffsetY` lowers the actual robot target from that surface, and `candidateMarkerLiftY` raises only the display marker. Removed the old shared AI marker lift field.
- Added fixed-height AI candidate marker display: `useFixedCandidateMarkerHeight` keeps marker X/Z aligned with the candidate while forcing marker Y to `candidateMarkerFixedWorldY`, fully decoupling marker height from grasp depth.
- Fixed fixed-base M9 pitch inversion by removing angle normalization before pitch clamping, so `invertFixedBasePitch` can actually flip the up/down direction.
- Added automatic finger-tip point lookup in `ScanManager` for transforms whose names include left/right and tip, reducing missing Inspector reference failures during grasp judgement.
- Removed the unsafe finger-tip auto-assignment fallback to `Left_Finger2` / `Right_Finger2`, because Blender bone/mesh origins can be far from the actual fingertip. The judge now requires explicit tip-point transforms or names containing left/right and tip.
- Changed the default candidate depth interpretation back to meters, because the AI response `candidate.depth` is expected to be a camera-distance value after server-side depth decoding; treating it again as inverted normalized depth pushed reconstructed candidates below the floor.
- Split fixed-base M9 pitch into servo-like pitch and Unity local joint pitch using `fixedBasePitchUnityAngleOffsetDeg`. The fixed-base log now reports `M9(servo/unity/clamped)` so servo-scale angles such as 160 degrees are not sent directly to the Unity elbow joint.

## 2026-07-10

- Investigated the Dex-Net-to-hand calibration path. Confirmed that `candidate.depth` is treated as a camera-distance value in meters, then `Camera.ViewportToWorldPoint` reconstructs the Unity world grasp point before conversion to the M9-origin hand coordinate system.
- Added `RobotHandController.fixedBaseFingerTipZOffsetMm` so the fixed-base M9 pitch calculation can use an effective hand-tip Z distance: `fixedBaseL3zMm + fixedBaseFingerTipZOffsetMm`. This keeps the base L3 value editable while allowing a finger-length or grasp-center extension to be added separately.
- Reworked the fixed-base M9 pitch calculation to use a direct geometric angle only: configured hand-tip vector angle minus target height angle, with `invertFixedBasePitch` as the remaining sign correction. The old servo/Unity pitch offsets are hidden from the Inspector and no longer affect the returned pitch.
- Added detailed fixed-base calibration diagnostics in `RobotHandController`: target world position, fixed-base origin, relative Unity vector, raw/adjusted M9 hand coordinates, configured L3 tip coordinate, measured finger-tip midpoint coordinate, optional control-point coordinate, configured-vs-measured deltas, target-vs-measured deltas, pitch radius, bias angle, target height angle, pre-invert pitch, inversion state, and clamped pitch.
- Added `RobotHandController.DebugWorldToFixedBaseExperiment` so other scripts can convert any Unity world point into the same M9-origin hand coordinate system used by fixed-base grasping.
- Added optional debug references in `RobotHandController`: `fixedBaseDebugLeftFingerTipPoint`, `fixedBaseDebugRightFingerTipPoint`, and `fixedBaseDebugControlPoint`.
- Added grasp calibration diagnostics in `ScanManager`. When a candidate is confirmed, the log now includes Dex-Net rank, image size, pixel center, depth, angle, Q value, depth mode/scale/offset, surface/grasp/marker world positions, grasp surface/world offsets, raw/adjusted hand coordinates, finger-tip world and hand coordinates, optional control-point world and hand coordinates, and target deltas.
- Added `ScanManager.graspControlPoint` as a submission/debug control point separate from the left/right finger-tip points. If assigned, the candidate-confirm and fixed-base logs show `controlPointExp`, `configuredMinusControlPointExp`, and `targetMinusControlPointExp`.
- Added plumbing from `ScanManager` to `RobotHandController` so the fixed-base debug logs use the same finger-tip/control-point references as the candidate-confirm logs.
- Added `RobotHandController.useFixedBaseControlPointForPitchReference`, disabled by default. When explicitly enabled, and when a `graspControlPoint` / `fixedBaseDebugControlPoint` is assigned, the fixed-base pitch calculation can use that point's current M9-coordinate `(X, Z)` as the initial hand-tip vector instead of only the configured L3 constants. For the submission path, leave this OFF so the motion depends on configured constants rather than Blender/Unity Transform positions.
- Extended fixed-base motion logs with `pitchRef` / `pitchReference`, showing whether pitch was calculated from the control point or the constant L3 values.
- Added `ScanManager.submissionSimpleCandidateJudge`, enabled by default, to bypass unreliable finger-tip Transform contact judgement for submission builds. When enabled, success judgement uses the object nearest the AI candidate position within `submissionCandidateJudgeRadius`, then runs the existing lift/score flow.
- Set `submissionCandidateJudgeRadius` default to `0.09` m. Increase it, for example to `0.12`, if the AI candidate is visually correct but the simple candidate judge misses nearby objects; decrease it if the game becomes too permissive.
- Calibration finding: Dex-Net candidate positions and M9 hand-coordinate conversion appear internally consistent, but Blender-authored hand visual geometry and Unity Transform positions can diverge significantly. Logged examples showed visible/control points placed by eye could produce hand-coordinate values such as `X ~= 0 mm` even when the intended control point should be near `X ~= -90 mm`.
- Calibration finding: using left/right finger-tip Transforms for both control and success judgement is unsafe until their Transform positions are proven to match the visible fingertip/grasp-center geometry. For the submission path, prefer constant L3-based fixed-base motion plus candidate-position-based success judgement.
- Operational note: `graspControlPoint` is mainly a debug/calibration reference. First confirm a candidate and check `Confirmed grasp calibration debug`; `controlPointExp` shows where that manually placed point actually is in M9-origin hand coordinates, but the submission motion does not depend on it while `useFixedBaseControlPointForPitchReference` is OFF.
- Operational note: when checking motion-time calibration, press Catch and read `Fixed-base calibration debug` before interpreting the hand motion. `pitchRef=constant` means the motion is using only `fixedBaseL3xMm` and `fixedBaseL3zMm + fixedBaseFingerTipZOffsetMm`; `pitchCalc` shows how that constant reference and the target height became the M9 pitch angle.
- Submission recommendation: leave `submissionSimpleCandidateJudge = true` and `useQThreshold = false` unless there is time to repair the Blender/Unity fingertip Transform alignment. This makes the demo playable based on the already reliable AI candidate position rather than unreliable visual fingertip Transforms.
- Added `SpawnedObjectPhysicsStabilizer` as an optional helper for submission tuning. Attach it next to `ObjectSpawner` to apply heavier mass, linear/angular drag, continuous collision detection, solver iteration settings, and high-friction/zero-bounce collider material to spawned objects so small contacts are less likely to scatter the pile.
- Tuning note: slow finger closing with `RobotHandController.fixedBaseCloseDuration`; use `SpawnedObjectPhysicsStabilizer` for object stability. This separates hand motion pacing from object physics damping.
